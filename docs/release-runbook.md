# Release Runbook

This document is the maintainer runbook for releasing `kyaukyuai/linear-cli` and recovering from publish failures across GitHub Releases, Homebrew, JSR, and npm.

## Scope

Use this runbook when:

- cutting a new release tag
- validating that publish automation completed
- recovering from partial publish failures
- checking the external service settings required for release automation

## Release Sources Of Truth

- version: `deno.json`
- packaged release metadata: `dist-workspace.toml`
- release notes: `CHANGELOG.md`
- git tag workflow: `justfile`
- generated GitHub Release workflow: `.github/workflows/release.yml`
- custom JSR/npm publish workflow: `.github/workflows/publish.yaml`
- cargo-dist build setup source: `.github/build-setup.yml`

`release.yml` is generated by cargo-dist. Do not hand-edit it. Update `dist-workspace.toml` or `.github/build-setup.yml`, then run `just dist-generate` or `dist generate`.

## External Service Prerequisites

### GitHub

Repository:

- `kyaukyuai/linear-cli`

Secrets:

- `HOMEBREW_TAP_TOKEN`
- `NPM_TOKEN`

Notes:

- `NPM_TOKEN` is no longer used for normal trusted publishing, but keep it available for manual fallback until the release flow has been stable for several releases.

### Homebrew

Tap repository:

- `kyaukyuai/homebrew-tap`

`release.yml` publishes formula updates to this tap.

### npm

Package:

- `@kyaukyuai/linear-cli`

Trusted publisher settings:

- organization or user: `kyaukyuai`
- repository: `linear-cli`
- workflow filename: `release.yml`
- environment name: leave blank

The package must remain public.

### JSR

Package:

- `@kyaukyuai/linear-cli`

Package settings:

- repository link must point to `kyaukyuai/linear-cli`
- description should stay populated
- compatible runtimes should stay accurate

## Standard Release Procedure

1. Make sure `main` is clean except for unrelated local files like `.agents/`.
2. Update `CHANGELOG.md`.
3. Update versions:
   - `deno.json`
   - `dist-workspace.toml`
4. Run validation locally:

```bash
deno check src/main.ts
deno fmt --check
deno lint
deno task test
deno task install
deno task generate-skill-docs
```

5. If release metadata or cargo-dist inputs changed, regenerate workflows:

```bash
just dist-generate
```

6. Commit the release bump.
7. Create and push the tag:

```bash
just tag
```

This pushes:

- `main`
- `vX.Y.Z`

## Expected Automation Flow

After the tag push:

1. `release.yml` creates the GitHub Release and publishes Homebrew.
2. `publish.yaml` publishes JSR on tag push.
3. `publish.yaml` publishes npm after the `Release` workflow completes successfully.

## Post-Release Verification Checklist

Check all of the following:

- GitHub Release exists for `vX.Y.Z`
- release assets include platform archives and `linear-npm-package.tar.gz`
- Homebrew formula in `kyaukyuai/homebrew-tap` updated
- JSR version exists: `https://jsr.io/@kyaukyuai/linear-cli`
- npm version exists:

```bash
npm view @kyaukyuai/linear-cli@X.Y.Z version
npm view @kyaukyuai/linear-cli dist-tags --json
```

## Failure Recovery

### GitHub Release succeeded, npm did not

First, inspect the `Publish` workflow logs.

If the version is already on npm, reruns are safe because `publish.yaml` skips existing versions.

If automation failed before npm publish and the version is not on npm, manually publish from the release asset:

```bash
gh release download "vX.Y.Z" -R kyaukyuai/linear-cli -p "linear-npm-package.tar.gz" -D /tmp/linear-npm --clobber
npm publish --access public /tmp/linear-npm/linear-npm-package.tar.gz
```

Re-check:

```bash
npm view @kyaukyuai/linear-cli@X.Y.Z version
```

### GitHub Release succeeded, JSR did not

First, inspect the `Publish` workflow logs.

If the error is repository authorization or package link related, verify the JSR package is linked to `kyaukyuai/linear-cli`.

If the version is not on JSR, do a manual publish from the repo checkout at the release commit:

```bash
npx jsr publish
```

Then verify:

- `https://jsr.io/@kyaukyuai/linear-cli`
- `https://jsr.io/@kyaukyuai/linear-cli/score`

### GitHub Release succeeded, Homebrew did not

Check the `publish-homebrew-formula` job in `release.yml`.

Verify:

- `HOMEBREW_TAP_TOKEN` is valid
- `kyaukyuai/homebrew-tap` exists and is writable
- the generated formula artifact exists in the release

If needed, rerun the failed release job after fixing access.

### Only part of the registries published

Do not cut a new tag just to recover a partial publish.

Preferred order:

1. Verify whether the version already exists on the failed registry.
2. If it exists, leave the rerun skipped.
3. If it does not exist, use the manual recovery path for that registry.
4. Keep the original `vX.Y.Z` tag.

## Operational Notes

- `publish.yaml` is designed to skip already-published JSR and npm versions.
- `release.yml` is generated. Edit the source files, not the generated file.
- `publish-npm` now runs after `Release` succeeds, so it should not poll for release assets anymore.
- If GitHub Actions warnings mention Node 20 deprecation again, check whether any third-party action reintroduced a `node20` runtime.

## When To Create A Follow-Up Issue

Create a new Linear issue if:

- a publish step requires repeated manual recovery
- a registry setting changed outside the repo
- a release workflow started depending on a newly deprecated GitHub Action runtime
- the runbook itself was insufficient to complete recovery
